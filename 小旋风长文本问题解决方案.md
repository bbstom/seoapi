# 小旋风长文本改写失败解决方案

## 🔍 问题现象

**症状：**
- ✅ 短文本（<100 字符）改写成功
- ❌ 长文本（>1000 字符）改写失败
- ✅ 服务器日志显示改写成功（errcode=0）
- ❌ 小旋风显示"改写失败"

**日志示例：**
```
[请求 xxx] ✓ 改写完成 - 总耗时: 16.52秒, 输出长度: 1261
[请求 xxx] 完整响应体长度: 1306 字符
[请求 xxx] ... (响应体太长，已截断)
[请求 xxx] ✅ 响应已发送
```

但小旋风显示：**改写失败**

## 🎯 根本原因

**小旋风对 API 响应体有大小限制！**

- 小旋风可以接收的响应体大小约为 **1300-1500 字符**
- 当改写后的文本 + JSON 结构超过这个限制时，小旋风无法正确解析
- 服务器端一切正常，问题出在小旋风客户端

**为什么短文本成功？**
- 短文本（27 字符）+ JSON 结构 ≈ 100 字符 ✅
- 长文本（1261 字符）+ JSON 结构 ≈ 1306 字符 ❌

## ✅ 解决方案

### 方案 1：限制采集文本长度（推荐）⭐

**在小旋风的采集规则中限制文本长度**

```
采集规则设置：
- 内容截取：前 800 字符
- 或使用正则：(.{1,800})
```

**优点：**
- ✅ 最简单有效
- ✅ 从源头控制文本长度
- ✅ 确保响应体不会超限
- ✅ 改写速度更快（文本短）

**推荐长度：**
- 800-1000 字符：安全范围
- 1000-1200 字符：可能成功
- >1200 字符：很可能失败

### 方案 2：使用 compact 参数压缩输出

**在小旋风的 post 格式中添加 `compact=1`**

```
post格式: txt={word}&mode=seo_fast&compact=1
```

**效果：**
- 移除多余的空格和换行
- 压缩响应体积
- 可以减少 10-20% 的大小

**示例：**

压缩前（1261 字符）：
```
@序章 传奇终结。

@天空被乌云遮蔽，光线黯淡的大地上满目疮痍。

螺旋向上的台阶，血污遍布的尸首...
```

压缩后（约 1100 字符）：
```
@序章 传奇终结。@天空被乌云遮蔽，光线黯淡的大地上满目疮痍。螺旋向上的台阶，血污遍布的尸首...
```

**注意：** 即使压缩，超长文本仍可能失败

### 方案 3：分段改写（适用于长文章）

**对于超长文本，分段处理：**

1. **在采集规则中分段**
   ```
   段落1：前 800 字符
   段落2：801-1600 字符
   段落3：1601-2400 字符
   ```

2. **分别调用 API 改写**
   - 每段独立改写
   - 小旋风自动合并结果

3. **最终拼接**
   - 在发布时合并所有段落

### 方案 4：使用更快的模型（减少超时风险）

**对于长文本，使用 Haiku 模型：**

```
post格式: txt={word}&mode=seo_fast&model=claude-haiku-4-5-20251001
```

**优点：**
- ✅ 速度快（3-5 秒 vs 15-20 秒）
- ✅ 成本低
- ✅ 减少超时风险

**注意：** 这不能解决响应体大小限制，但可以提高成功率

## 📝 常见问题

### Q1: 为什么服务器显示成功，小旋风显示失败？

**A:** 服务器确实成功了，但小旋风无法接收超过 1300-1500 字符的响应体。这是小旋风客户端的限制，不是服务器问题。

### Q2: compact=1 能解决所有长文本问题吗？

**A:** 不能。compact=1 只能减少 10-20% 的大小。如果原文本超过 1400 字符，压缩后仍可能超限。最好的方法是限制采集长度。

### Q3: 可以修改小旋风的响应体大小限制吗？

**A:** 不能。这是小旋风软件的内部限制，无法修改。只能从采集端控制文本长度。

### Q4: 分段改写会影响文章质量吗？

**A:** 可能会。分段改写后，段落之间的连贯性可能受影响。建议在采集阶段就控制长度，而不是分段改写。

### Q5: 为什么有时候 1300 字符成功，有时候失败？

**A:** 小旋风的限制可能不是固定的，可能受网络、系统资源等因素影响。建议保持在 1200 字符以下以确保稳定性。

**最简单有效的解决方案：**

1. ⭐ **在小旋风采集规则中限制文本长度为 800-1000 字符**

3. 使用 `seo_fast` 模式提高速度
4. 监控服务器日志中的长度警告

- 控制源头（采集长度）比压缩输出更有效

**配置示例：**
```

```
