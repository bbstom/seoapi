# 故障转移功能说明

## 功能概述

文本改写接口支持智能故障转移功能。根据用户的选择，系统会采用不同的策略：

- **用户手动指定节点**：精确控制，禁用故障转移
- **用户使用默认配置**：智能容错，启用故障转移

## 核心原则

**用户手动指定 = 精确控制，不自动切换**
**用户使用默认 = 智能容错，自动故障转移**

---

## 工作模式

### 模式1：用户手动指定节点（禁用故障转移）

**触发条件：**
- 前端选择了具体的 API 节点（不是"使用默认节点"）
- 请求参数包含 `config_id`

**行为：**
- 只使用用户指定的节点
- 如果该节点失败，直接返回错误
- 不会自动切换到其他节点

**日志示例：**
```
[请求 abc123] 用户手动指定节点 ID: 2
[请求 abc123] ✓ 使用用户指定节点: fucaixie.xyz
[请求 abc123] 用户手动指定节点，禁用故障转移
[请求 abc123] 可用节点数: 1, 故障转移: 禁用
```

**使用场景：**
- 测试特定节点
- 使用特定节点的特殊功能
- 调试节点问题
- 需要精确控制使用哪个节点

---

### 模式2：使用默认配置（启用故障转移）

**触发条件：**
- 前端选择"使用默认节点"
- 请求参数不包含 `config_id`

**行为：**
- 先使用默认节点
- 如果默认节点失败，自动尝试其他可用节点
- 按节点创建时间倒序尝试
- 记录故障转移日志

**日志示例：**
```
[请求 abc123] 使用负载均衡模式
[请求 abc123] 尝试使用默认外部API配置 ID: 1
[请求 abc123] ✓ 成功加载外部API配置: api123
[请求 abc123] 使用默认配置，启用故障转移
[请求 abc123] 可用节点数: 3, 故障转移: 启用

[请求 abc123] 尝试节点 1/3: api123
[请求 abc123] 节点 api123 失败: 连接超时
[请求 abc123] 准备尝试下一个节点...
[请求 abc123] 尝试节点 2/3: fucaixie.xyz
[请求 abc123] ✅ 故障转移成功: api123 → fucaixie.xyz
```

**使用场景：**
- 日常使用（推荐）
- 需要高可用性
- 不关心具体使用哪个节点
- 希望系统自动处理故障

---

## 节点选择优先级

### 1. 用户手动指定节点（前端传递 config_id）
```
前端传递 config_id → 使用该节点 → 禁用故障转移
```

### 2. 令牌固定节点
```
令牌配置 nodeStrategy=fixed → 使用固定节点 → 禁用故障转移
```

### 3. 用户默认配置
```
用户设置 defaultConfigId → 使用默认节点 → 启用故障转移
```

### 4. 标记为默认的配置
```
api_configs.is_default=true → 使用该节点 → 启用故障转移
```

### 5. 个人中心配置（旧系统）
```
users.claudeApiKey → 使用个人配置 → 无故障转移
```

## 前端界面对应

### 改写配置面板

```
┌─────────────────────────────────────┐
│ API 节点: [使用默认节点 ▼]          │  ← 启用故障转移
│           [api123 ▼]                │  ← 禁用故障转移
│           [fucaixie.xyz ▼]          │  ← 禁用故障转移
└─────────────────────────────────────┘
```

- **选择"使用默认节点"**：系统会自动选择最佳节点，失败时自动切换
- **选择具体节点**：只用这个节点，失败就失败，不会切换

## 使用示例

### 场景1：用户选择 fucaixie.xyz 节点（禁用故障转移）

**前端操作：**
```javascript
{
  text: "测试文本",
  config_id: 2,  // fucaixie.xyz 的 ID
  model: "claude-sonnet-4-6"
}
```

**后端行为：**
1. 检测到 `config_id=2`
2. 使用 fucaixie.xyz 节点
3. 禁用故障转移
4. 如果失败，直接返回错误给用户

**结果：**
- ✅ 成功：返回改写结果
- ❌ 失败：返回错误，不会切换到 api123

---

### 场景2：用户选择"使用默认节点"（启用故障转移）

**前端操作：**
```javascript
{
  text: "测试文本",
  // 没有 config_id
  model: "claude-sonnet-4-6"
}
```

**后端行为：**
1. 没有 `config_id`，使用默认配置
2. 假设默认是 api123
3. 启用故障转移
4. 如果 api123 失败，自动尝试 fucaixie.xyz

**结果：**
- ✅ api123 成功：返回改写结果
- ⚠️ api123 失败 → 自动切换到 fucaixie.xyz
  - ✅ fucaixie.xyz 成功：返回改写结果
  - ❌ fucaixie.xyz 失败：继续尝试下一个节点

---

### 场景3：所有节点都失败（仅在启用故障转移时）

```
[请求 abc123] 尝试节点 1/2: fucaixie.xyz
[请求 abc123] 节点 fucaixie.xyz 失败: 连接超时
[请求 abc123] 尝试节点 2/2: api123
[请求 abc123] 节点 api123 失败: 连接超时
[请求 abc123] ❌ 所有节点都失败了，已尝试 2 个节点
```

## 配置建议

### 1. 日常使用（推荐）

**推荐配置：**
- 设置一个稳定的默认节点
- 添加 2-3 个备用节点
- 前端选择"使用默认节点"

**优点：**
- 自动故障转移，高可用
- 无需手动切换节点
- 系统自动选择最佳节点

### 2. 测试调试

**推荐配置：**
- 前端手动选择具体节点
- 观察该节点的行为
- 不会被故障转移干扰

**优点：**
- 精确控制使用哪个节点
- 便于测试和调试
- 不会混淆节点来源

### 3. 生产环境

**推荐配置：**
- 配置多个高质量节点
- 设置最稳定的为默认
- 启用故障转移
- 定期查看故障转移历史

**优点：**
- 最大化可用性
- 自动处理节点故障
- 用户无感知切换

## 常见问题

### Q1: 我选择了 fucaixie.xyz，为什么用的是 api123？

**A:** 这种情况不应该发生！检查以下情况：

1. **前端是否真的传递了 config_id？**
   - 打开浏览器开发者工具 → Network
   - 查看 `/api/rewrite` 请求
   - 检查请求体是否包含 `config_id`

2. **fucaixie.xyz 是否配置正确？**
   - 进入"AI 节点管理"
   - 编辑 fucaixie.xyz 节点
   - 点击"测试连接"验证

3. **查看服务器日志**
   - 搜索 "用户手动指定节点"
   - 如果没有这条日志，说明前端没有传递 config_id

**注意：** 如果你手动选择了节点，系统不会自动切换到其他节点。如果该节点失败，会直接返回错误。

### Q2: 如何完全禁用故障转移？

**A:** 有两种方法：

1. **前端始终选择具体节点**（推荐）
   - 不选择"使用默认节点"
   - 选择具体的节点名称

2. **只配置一个节点**
   - 删除或禁用其他节点
   - 只保留一个启用的节点

### Q3: 故障转移会影响性能吗？

**A:** 
- **正常情况**：不会，直接使用第一个节点
- **故障情况**：会增加延迟（每次重试约 1-3 秒）
- **建议**：确保默认节点稳定，减少故障转移次数

### Q4: 如何查看是否发生了故障转移？

**A:** 三种方法：

1. **服务器日志**
   ```
   [请求 xxx] ✅ 故障转移成功: api123 → fucaixie.xyz
   ```

2. **前端监控仪表板**
   - 监控仪表板 → 故障转移历史

3. **数据库查询**
   ```sql
   SELECT * FROM failover_logs 
   WHERE user_id = 1 
   ORDER BY failover_time DESC 
   LIMIT 10;
   ```

### Q5: 为什么要区分手动指定和默认配置？

**A:** 这是为了给用户更好的控制：

- **手动指定**：用户明确知道要用哪个节点，可能是为了测试、调试或使用特定功能
- **默认配置**：用户只关心结果，不关心用哪个节点，希望系统自动处理故障

## 技术细节

### 数据库表结构

```sql
CREATE TABLE failover_logs (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id INT NOT NULL,
    failover_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    from_node_id INT,
    to_node_id INT NOT NULL,
    failure_reason VARCHAR(500),
    success BOOLEAN,
    retry_count INT,
    total_latency INT,
    ...
);
```

### API 调用流程

```javascript
// 1. 检查是否手动指定节点
if (config_id) {
    // 用户手动指定，禁用故障转移
    const config = await getConfig(config_id);
    return await callAI(config); // 失败就失败
}

// 2. 使用默认配置，启用故障转移
const allConfigs = [默认节点, 备用节点1, 备用节点2];

// 3. 依次尝试
for (const config of allConfigs) {
    try {
        const result = await callAI(config);
        return result; // 成功
    } catch (error) {
        // 记录错误，继续下一个
    }
}

// 4. 所有节点都失败
throw new Error('所有节点都失败');
```

## 总结

| 场景 | config_id | 故障转移 | 行为 |
|------|-----------|----------|------|
| 手动选择节点 | ✅ 有 | ❌ 禁用 | 只用指定节点，失败就失败 |
| 使用默认节点 | ❌ 无 | ✅ 启用 | 自动选择，失败自动切换 |
| 令牌固定节点 | ❌ 无 | ❌ 禁用 | 只用令牌节点，失败就失败 |

**核心理念：**
- 用户明确指定 = 尊重用户选择，不自动切换
- 用户使用默认 = 智能容错，自动故障转移
